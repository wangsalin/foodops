# FoodOps Community 开源发布文案

## 推荐标题

我们开源了一个面向连锁餐饮门店的本地运营系统：FoodOps Community

## 正文

过去一段时间，我们一直在思考一个问题：中小型连锁餐饮团队，如果还没有完整的私有化系统、企业集成和数据中台，能不能先用一套更轻的方式，把门店运营的基本动作跑起来？

很多团队真正卡住的，不一定是复杂算法或大平台能力，而是每天反复发生的基础工作：

- 门店、商品、物料、人员这些基础数据分散在不同表里。
- 销售、库存、评价等数据可以导出，但很难统一看。
- 异常发现以后，靠群消息、截图、人工催办推进。
- 门店反馈完成情况后，过程记录和责任链路不清楚。
- 后续复盘时，很难知道一个问题是如何发现、派发、处理和关闭的。

所以我们把这条最小运营链路整理出来，做成了开源项目：

[FoodOps Community](https://github.com/wangsalin/foodops)

## 它解决什么

FoodOps Community 不是一个“大而全”的餐饮平台，也不试图替代外卖平台、ERP 或企业内部系统。

它先聚焦一条更小、更容易自托管的运营链路：

1. 维护门店、商品、物料、人员等主数据。
2. 手动导入销售、商品销售、库存、评价等业务数据。
3. 在 dashboard 里看门店和商品表现。
4. 用本地规则生成运营预警。
5. 把预警转换成任务，分配负责人和截止时间。
6. 门店通过 H5 页面反馈处理结果。
7. 系统保留通知、任务历史和审计日志，方便复盘。

这套链路的目标很明确：先让一个团队能在本地跑起“数据导入 -> 预警 -> 任务 -> 反馈 -> 复盘”的日常动作。

## 为什么先开源社区版

我们把社区版做得比较克制，只保留本地运营核心。

社区版包含：

- 主数据管理
- 手动导入
- dashboard
- 本地规则预警
- 任务派发
- H5 反馈
- 站内通知
- 审计日志
- demo seed

社区版不包含：

- 企业微信、飞书等企业连接器
- 客户专属 SSO
- 外部 AI provider
- 舆情采集、社媒工作流
- 私有客户数据和私有品牌资产
- 客户定制自动化

这样做是为了让开源项目边界清楚。社区版适合讨论通用运营模型、基础数据结构、导入模板、任务流和可自托管架构；企业连接器和客户定制能力应该留在插件或企业版边界里。

## 适合谁参与

我们希望第一批参与者来自几类人：

- 做过餐饮、零售、门店运营系统的开发者。
- 熟悉 FastAPI、Next.js、PostgreSQL 的全栈开发者。
- 对自托管 SaaS、行业工具、运营中台感兴趣的人。
- 有真实门店运营经验，愿意帮忙校准业务流程的人。

现在仓库里已经准备了第一批 good first issues：

- 添加销售、商品销售、库存、评价导入模板。
- 给导入 normalizer 增加测试。
- 给 alert-to-task 流程增加测试。
- 给 dashboard、alerts、tasks、H5 反馈增加前端 smoke test。
- 丰富 demo seed，让本地启动后能看到更完整的业务链路。

## 项目地址

GitHub:

https://github.com/wangsalin/foodops

如果你对连锁餐饮运营、行业 SaaS、自托管工具或开源协作感兴趣，可以先 star、跑一下本地 demo，或者直接从 issue 里挑一个小任务。

我们更希望第一波反馈是具体的：

- 哪一步跑不起来？
- 哪个导入字段不合理？
- 哪个页面不符合门店运营习惯？
- 哪个 good first issue 还不够清楚？

这些反馈比泛泛的鼓励更有价值。
