## 正文

我们把 FoodOps Community 开源了。

GitHub： https://github.com/wangsalin/foodops

这是一个面向连锁餐饮、茶饮、烘焙等多门店团队的本地经营系统。它不是外卖平台，也不是要替代 ERP，而是先解决一个很实际的问题：

很多门店经营动作并不是没有数据，而是数据散在 POS、外卖后台、表格、群消息和人工巡店记录里。总部看到问题以后，还要靠人去截图、催办、跟进、复核，最后很难沉淀成一条完整链路。

FoodOps Community 想先把这条链路跑起来：

1. 导入门店经营数据  
   早期不强依赖平台 API，可以先用 CSV/XLSX 导入销售、库存、评价等数据，适合做内部试点和开发验证。

2. 生成经营看板  
   把营收、订单、预警、任务闭环、导入批次等关键状态集中到一个 dashboard，方便总部快速看出今天哪里不对。

3. 规则触发异常预警  
   例如库存低于安全线、评价异常、销售波动、门店执行异常等，先用可解释的规则跑通第一版。

4. 预警转任务  
   总部可以把预警派成任务，指定负责人，跟踪确认、处理、反馈、审核状态。

5. 门店 H5 反馈  
   门店负责人不一定需要登录完整后台，可以通过 H5 链接提交处理结果、备注和图片。

6. 审计和通知留痕  
   每个关键动作都有记录，后面才能做复盘、追责、绩效和自动化优化。

我们现在更希望它成为一个开发者可以一起参与的开源项目，而不是关起门来做一个“大而全”的系统。

当前仓库里已经有：

- Next.js + Ant Design 前端
- FastAPI 后端
- PostgreSQL / Redis 本地环境
- Alembic 数据库迁移
- demo seed 数据
- 权限、认证、审计、预警、任务、导入等基础模块
- 真实页面截图和启动文档

接下来最需要开发者一起做的方向：

- 把本地部署流程继续打磨到更顺滑
- 增加更多导入模板和校验规则
- 优化预警规则引擎，让规则更容易配置
- 完善任务/H5 反馈体验
- 增加测试覆盖和 CI 稳定性
- 做更多适合中国餐饮门店的真实运营场景
- 后续探索 AI 摘要、智能派单、经营建议，但不急着把 AI 做成噱头

我的判断是，餐饮经营系统如果想真正有生命力，不能只靠一个团队闭门造车。开发者、门店经营者、区域督导、数据同学都应该能参与进来，把真实问题沉淀成 issue、模板、规则和代码。

如果你对开源 SaaS、餐饮数字化、本地优先部署、经营数据自动化、AI agent 在真实业务里的落地感兴趣，欢迎来看看仓库：

https://github.com/wangsalin/foodops

可以从这几件事开始：

- 跑一下本地环境，看看文档哪里不顺
- 提一个真实餐饮场景 issue
- 认领 good first issue
- 帮忙补一个导入模板
- 帮忙优化一个页面或一个接口
- 直接 fork 做你认为应该有的能力

FoodOps Community 现在还早，但方向很清楚：先把“数据 -> 预警 -> 任务 -> 门店反馈 -> 总部复核 -> 审计沉淀”这条链路做扎实。

欢迎一起开发，一起使用，一起把它变成一个真正可落地的开源餐饮经营系统。
